# PBIP Markdown Bridge
Static web app (GitHub Pages ready) that converts:
- **Power BI PBIP ZIP → single Markdown package**
- **Markdown package → PBIP ZIP**

This is meant for AI-assisted editing workflows where PBIP projects are exchanged as one text file.

## Project structure
- `index.html` — main app UI
- `converter.js` — conversion + validation core
- `app.js` — UI orchestration
- `test/roundtrip.html` — browser roundtrip test harness
- `.github/workflows/pages.yml` — GitHub Pages deploy workflow

## Usage
1. Open the site (`index.html` locally or hosted on GitHub Pages).
2. ZIP → Markdown:
   - Upload your PBIP ZIP (for example `SSXYZ_ActiveLots.zip`)
   - Click **Convert ZIP to Markdown**
   - Download the generated `.md` package
3. Edit Markdown with your AI agent.
4. Markdown → ZIP:
   - Upload the edited `.md`
   - Click **Convert Markdown to ZIP**
   - Download rebuilt `.zip`

## Markdown package format (v1)
The file includes:
1. Metadata comment block
2. Folder structure tree
3. File index
4. One file block per project file

Metadata marker:
```text
<!-- PBIPMD:METADATA
{ ...json... }
PBIPMD:METADATA -->
```

File block format:
- Header: `### FILE \`relative/path.ext\``
- Metadata lines:
  - `- encoding: utf8|base64`
  - `- bytes: <integer>`
  - `- sha256: <64-hex>`
- Followed by a fenced payload block (`text` for UTF-8 or `base64` for binary), then closing fence.

## Validation rules
On Markdown → ZIP conversion, the app rejects:
- missing/invalid metadata
- duplicate file paths
- unsafe paths (absolute paths, drive roots, `..`, empty segments)
- malformed file headers
- invalid base64 payloads
- byte size mismatches
- SHA-256 mismatches

## Notes on transient files
The UI includes a toggle for transient Power BI artifacts:
- `.pbi/cache.abf`
- `.pbi/localSettings.json`
- `.pbi/editorSettings.json`

Keep included for strict fidelity; exclude to reduce AI noise.

## AI editing guidance
When editing package Markdown:
- Keep every `### FILE` header intact
- Keep `encoding`, `bytes`, `sha256` lines (they can be regenerated by reconversion, but parser requires structure)
- Do not rename or duplicate paths unless intentional
- Do not alter metadata marker names (`PBIPMD:METADATA`)

## Local testing
Open:
- `index.html` for main app
- `test/roundtrip.html` for roundtrip validation

No build step required.
For a CLI verification test case:
1. `npm install`
2. `npm run test:roundtrip`

## GitHub Pages deployment
1. Push this folder as its own repository.
2. Ensure Actions are enabled.
3. The included workflow deploys static files from repo root to Pages.

If needed, add a `.nojekyll` file (already included) to avoid Jekyll processing.
